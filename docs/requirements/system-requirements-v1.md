# Системные требования (v1)

## Интеграция АС КОДА с AI-агентом RAIN для формирования БТ

**Бизнес-требования**

```
Цель:
Обеспечить автоматизированное формирование бизнес-требования в Confluence из карточки завершенной симуляции без перехода пользователя в другую систему.

Бизнес-правила:
- Функциональность доступна только для завершенных симуляций.
- Первая итерация поддерживает продукт "Кредитные карты".
- Перед публикацией БТ требуется явное подтверждение пользователя.
- При наличии ранее созданного БТ пользователь может инициировать создание нового.

Ограничения:
- Интеграция реализуется через API /dialog/init и /dialog/{sessionId}/continue.
- На уровне ИФТ в Q2 допускается временная работа без OTT.
- Сессия диалога хранится 30 минут с момента последней активности.
```

**Пользовательские требования к АС КОДА**

- Отображать кнопку "Сформировать БТ" на вкладке "Основная информация" для симуляций в статусе "Завершена".
- По нажатию кнопки открывать диалог с агентом и инициализировать сессию с передачей контекста симуляции.
- Поддерживать перезапуск сессии через кнопку "Перезапустить сессию".
- Отображать сообщения агента в интерфейсе и сохранять итоговую ссылку на БТ в поле "Ссылка на БТ" вкладки "Итоги симуляции".

**Критерии приемки**

```
1. Если статус симуляции "Завершена", кнопка "Сформировать БТ" отображается; иначе не отображается.
2. После запуска диалога АС КОДА отправляет POST /dialog/init с sessionId и contextPrompt, содержащим обязательные поля.
3. Если от агента получен btUrl, ссылка сохраняется в "Ссылка на БТ" и отображается как кликабельная.
```

## Функциональные требования

### Реализация для FRONTEND

**Описание UI**

| Экран | Результат |
| --- | --- |
| Карточка симуляции, вкладка "Основная информация" | Для статуса "Завершена" отображается кнопка "Сформировать БТ". |
| Модальное окно диалога с агентом | Отображаются сообщения, поле ввода, кнопка "Перезапустить сессию". |
| Карточка симуляции, вкладка "Итоги симуляции" | Поле "Ссылка на БТ" содержит ссылку на последний созданный документ. |

```
FE-1. Реализовать условный рендеринг кнопки "Сформировать БТ" по статусу симуляции.
FE-2. При открытии диалога формировать sessionId (UUID), отправлять init-запрос и отображать первый ответ агента.
FE-3. Передавать каждое пользовательское сообщение в POST /dialog/{sessionId}/continue (userPrompt).
FE-4. Поддержать отображение промежуточного состояния загрузки, ошибок и механизма повторных попыток (до 3).
FE-5. Реализовать кнопку "Перезапустить сессию": генерация нового sessionId, повторный POST /dialog/init с тем же contextPrompt, очистка истории.
FE-6. Реализовать хранение истории диалога 30 минут с момента последнего сообщения и восстановление при повторном открытии.
FE-7. При получении btUrl выполнить сохранение ссылки в АС КОДА и обновить UI поля "Ссылка на БТ".
FE-8. Если contextPrompt.bt=true, в первом сообщении диалога отразить наличие ранее созданного БТ и запрос на подтверждение создания нового.
```

**Реализация BACKEND**

```
BE-1. Предоставить endpoint POST /dialog/init:
- принимать sessionId и contextPrompt;
- валидировать структуру и допустимые значения (например, product);
- возвращать HTTP 400 при ошибках валидации с текстовым описанием;
- возвращать первый agentResponse с вопросом о необходимости создания БТ.

BE-2. Предоставить endpoint POST /dialog/{sessionId}/continue:
- принимать userPrompt;
- использовать ранее накопленный контекст сессии;
- при отсутствии sessionId возвращать HTTP 404;
- возвращать очередной agentResponse или финальный результат.

BE-3. Реализовать state-machine диалога:
- запрос недостающих полей (businessGoal, customer, riskParam и др.);
- обязательный этап превью БТ перед публикацией;
- обязательный этап явного подтверждения Да/Нет;
- завершение без публикации при ответе "Нет".

BE-4. Реализовать интеграцию с Confluence:
- создание страницы по шаблону "ПРМ КК";
- заполнение обязательных разделов: дата, название изменений, бизнес-цель, бизнес-эффект, заказчик, аналитик, ограничения, таблица AS IS/TO BE;
- возвращать btUrl при успехе, ошибку без btUrl при неуспехе.

BE-5. Реализовать хранение сессий:
- TTL 30 минут с момента последней активности;
- по истечении TTL сессия недоступна, история не восстанавливается.

BE-6. Реализовать защищенное взаимодействие:
- HTTPS-only;
- mTLS для межсервисной аутентификации;
- OTT для авторизации вызовов (кроме временного исключения на ИФТ в Q2).

BE-7. Реализовать аудит и журналирование:
- фиксировать sessionId, simulationId, ФИО, HTTP-код ответа, результат создания БТ (btUrl/ошибка);
- логировать повторные попытки и финальный статус операции.
```

|  |  |
| --- | --- |
| **Сервис** | AI-агент RAIN + интеграционный слой АС КОДА |
| ИФТ URL | Определяется окружением деплоя (IFT/PSI/PROD) |
| База данных | Хранилище сессий диалога (TTL 30 минут) |
| Методы | `POST /dialog/init`, `POST /dialog/{sessionId}/continue` |

**Изменения в ролевой модели**

- Использовать существующую роль `experiment_editor_CC` для доступа к запуску сценария "Сформировать БТ".
- Новые роли не требуются.
- При добавлении новых API-методов или расширении прав доступа требуется отдельная актуализация страницы ролевой модели.

**Отправка в Аудит**

- Аудит обязателен для событий:
- запуск сессии формирования БТ;
- перезапуск сессии;
- создание страницы в Confluence;
- завершение с ошибкой после исчерпания retry.

**BUGS (Если есть)**

- На этапе ИФТ (Q2) возможно временное отключение OTT; требуется регистрация и контроль технического долга до вывода в ПСИ/ПРОМ.
- Требуется уточнение SLA по таймауту ответа агента для унификации поведения кнопки "Перезапустить сессию".
