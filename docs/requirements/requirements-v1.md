# Документ требований

## Введение

Документ описывает требования к интеграции АС КОДА с AI-агентом RAIN для формирования бизнес-требований (БТ) в Confluence после завершения симуляции.

Диалог с агентом запускается со страницы просмотра симуляции, во вкладке «Основная информация». Пользователь нажимает кнопку «Сформировать БТ», после чего АС КОДА передаёт контекст симуляции в RAIN, а агент ведёт диалог, формирует БТ по шаблону «ПРМ КК» и возвращает ссылку на созданную страницу.

Первая итерация (Q2) ограничена продуктом «Кредитные карты».

---

## Термины и сокращения

- **АС КОДА** — система, из интерфейса которой запускается диалог с агентом.
- **Агент RAIN** — сервис, который ведёт диалог и формирует БТ.
- **БТ** — бизнес-требование, создаваемое в Confluence по шаблону «ПРМ КК».
- **ПРМ аналитик** — пользователь с ролью `experiment_editor_CC`.
- **sessionId** — идентификатор сессии диалога (UUID), генерируется АС КОДА.
- **simulationId** — идентификатор симуляции (UUID, поле «ID ФП Симуляции»).
- **contextPrompt** — объект с данными симуляции.
- **СУДИР** — источник ФИО пользователя.
- **ИФТ / ПСИ / ПРОМ** — тестовый, предсерийный и промышленный стенды.

---

## Требования

### Требование 1. Отображение кнопки запуска

**Пользовательская история:**  
Как ПРМ аналитик, я хочу видеть кнопку «Сформировать БТ» на странице завершённой симуляции, чтобы начать создание БТ без перехода в другую систему.

#### Критерии приёмки

1. Если симуляция имеет статус «Завершена», АС КОДА отображает кнопку «Сформировать БТ» на странице симуляции (вкладка «Основная информация»).
2. Если статус симуляции отличается от «Завершена», кнопка «Сформировать БТ» не отображается.
3. В открытом диалоге с агентом отображается кнопка «Перезапустить сессию».

---

### Требование 2. Проверка существующего БТ

**Пользовательская история:**  
Как ПРМ аналитик, я хочу получать предупреждение о наличии уже созданного БТ, чтобы не создать дублирующий документ случайно.

#### Критерии приёмки

1. После нажатия «Сформировать БТ» АС КОДА проверяет поле «Ссылка на БТ» во вкладке «Итоги симуляции».
2. Если ссылка есть, в `contextPrompt.bt` передаётся `true`.
3. Если ссылки нет, в `contextPrompt.bt` передаётся `false`.

---

### Требование 3. Формирование и передача контекста

**Пользовательская история:**  
Как ПРМ аналитик, я хочу, чтобы агент получал максимум данных автоматически, чтобы не вводить их вручную.

#### Критерии приёмки

1. АС КОДА при запуске диалога отправляет `POST /dialog/init` с полями `sessionId`, `contextPrompt`.
2. В `contextPrompt` передаются: `simulationId`, `product`, `date`, `name`, `user`, `businessEffect`, `bt`, `riskParam`.
3. Значение `date` равно дате открытия диалогового окна.
4. Значение `name` по умолчанию: «Изменение Риск-параметров».
5. Поля `riskParam[*].name`, `riskParam[*].asIs`, `riskParam[*].toBe` берутся из вкладки «Изменение Риск-параметров»: «Риск-параметр», «Текущее значение», «Новое значение».
6. В `riskParam` включаются все риск-параметры, в том числе с одинаковыми `asIs` и `toBe`.
7. Если часть данных отсутствует (например, `user`, `businessEffect`, `simulationId`), агент запрашивает недостающее у пользователя в ходе диалога.
8. Для `asIs` и `toBe` поддерживаются типы `number`, `string`, `boolean`.

---

### Требование 4. Первый ответ агента

**Пользовательская история:**  
Как ПРМ аналитик, я хочу быстро получить первый ответ агента, чтобы не прерывать работу.

#### Критерии приёмки

1. После получения `POST /dialog/init` агент возвращает `agentResponse`.
2. В первом сообщении агент задаёт вопрос о необходимости создания БТ.
3. Ответ агента отображается пользователю в диалоговом окне.
4. При ошибке проверки входных данных агент возвращает HTTP 400 и понятное описание ошибки.

---

### Требование 5. Логика диалога

**Пользовательская история:**  
Как ПРМ аналитик, я хочу, чтобы агент задавал только нужные уточняющие вопросы.

#### Критерии приёмки

1. Ответ пользователя передаётся в `POST /dialog/{sessionId}/continue` в поле `userPrompt`.
2. Агент запрашивает только отсутствующие или пустые данные.
3. Агент запрашивает «Бизнес-цель» и «Кто заказчик», если эти поля не были получены ранее.
4. Если `contextPrompt.riskParam` не передан или пуст, агент запрашивает список изменений в формате: название, AS IS, TO BE.
5. Если пользователь отвечает «Нет» на вопрос о создании БТ, агент завершает диалог без создания страницы.

---

### Требование 6. Подтверждение перед публикацией

**Пользовательская история:**  
Как ПРМ аналитик, я хочу проверять итоговую версию БТ перед публикацией.

#### Критерии приёмки

1. После сбора данных агент показывает пользователю итоговую структуру БТ.
2. Перед созданием страницы агент обязательно запрашивает явное подтверждение («Да»/«Нет»).
3. Без подтверждения пользователя страница в Confluence не создаётся.
4. БТ содержит как минимум: дату создания, наименование изменений, бизнес-цель, бизнес-эффект, заказчика, ответственного аналитика, ограничения, таблицу риск-параметров AS IS / TO BE.

---

### Требование 7. Создание БТ и возврат ссылки

**Пользовательская история:**  
Как ПРМ аналитик, я хочу, чтобы после подтверждения БТ создавался автоматически.

#### Критерии приёмки

1. После подтверждения пользователя агент создаёт страницу в Confluence по шаблону «ПРМ КК».
2. При успешном создании агент возвращает `btUrl` (ссылку на страницу).
3. При ошибке создания агент возвращает сообщение об ошибке без `btUrl`.
4. Если пользователь подтвердил создание нового БТ при `contextPrompt.bt = true`, создаётся новая страница.

---

### Требование 8. Сохранение ссылки в АС КОДА

**Пользовательская история:**  
Как ПРМ аналитик, я хочу видеть ссылку на созданный БТ в итогах симуляции.

#### Критерии приёмки

1. Если от агента получен непустой `btUrl`, АС КОДА сохраняет его в поле «Ссылка на БТ» вкладки «Итоги симуляции».
2. Если БТ несколько, отображается ссылка на последний созданный документ.
3. Ссылка отображается как кликабельный элемент.

---

### Требование 9. Управление сессией диалога

**Пользовательская история:**  
Как ПРМ аналитик, я хочу вернуться к диалогу в течение 30 минут без потери переписки.

#### Критерии приёмки

1. АС КОДА хранит состояние диалога (историю сообщений) 30 минут с момента последнего сообщения.
2. Если пользователь повторно открывает диалог в пределах 30 минут, восстанавливается предыдущая переписка.
3. Если прошло более 30 минут бездействия, сессия закрывается, история не восстанавливается.
4. По кнопке «Перезапустить сессию» формируется новый `sessionId`, отправляется новый `POST /dialog/init` с тем же `contextPrompt`.
5. После перезапуска диалог начинается заново, без предыдущей истории.

---

### Требование 10. Обработка ошибок и повторные попытки

**Пользовательская история:**  
Как ПРМ аналитик, я хочу, чтобы временные ошибки обрабатывались автоматически.

#### Критерии приёмки

1. Если ответ агента имеет HTTP-код, отличный от 200, АС КОДА выполняет до 3 повторных попыток.
2. Если все попытки неуспешны, пользователю предлагается «Перезапустить сессию».
3. Если агент не отвечает в пределах допустимого времени ожидания, кнопка «Перезапустить сессию» становится доступной.
4. В журнале фиксируются: `sessionId`, `simulationId`, ФИО, HTTP-код, результат создания БТ (`btUrl` или ошибка).

---

### Требование 11. Защита взаимодействия

**Пользовательская история:**  
Как администратор, я хочу, чтобы обмен данными между АС КОДА и агентом был защищён.

#### Критерии приёмки

1. Взаимодействие выполняется только по HTTPS.
2. Аутентификация выполняется по двусторонним TLS-сертификатам и OTT.
3. Для стенда ИФТ в Q2 допускается временная работа без OTT при фиксации технического долга.
4. Трафик идёт через сетевую зону iAZ.

---

### Требование 12. Проверка входных данных API

**Пользовательская история:**  
Как разработчик, я хочу получать понятные ошибки валидации.

#### Критерии приёмки

1. Если в `POST /dialog/init` отсутствует `sessionId`, агент возвращает HTTP 400 и описание ошибки.
2. Если `contextPrompt.product` содержит недопустимое значение, агент возвращает HTTP 400 с перечнем допустимых значений.
3. Если в `POST /dialog/{sessionId}/continue` передан несуществующий `sessionId`, агент возвращает HTTP 404.
4. Поля `riskParam[*].asIs` и `riskParam[*].toBe` принимают типы `number`, `string`, `boolean`.

---

### Требование 13. Очередность сообщений в диалоге (MVP)

**Пользовательская история:**  
Как ПРМ аналитик, я хочу, чтобы диалог работал предсказуемо и без потери контекста, когда общение с агентом идет через REST.

#### Критерии приёмки

1. В MVP АС КОДА работает в режиме «строго по очереди»: допускается только один активный запрос `POST /dialog/{sessionId}/continue` на сессию.
2. После отправки сообщения поле ввода и кнопка отправки блокируются до получения ответа агента или наступления таймаута.
3. При блокировке интерфейс показывает явный статус ожидания ответа агента.
4. При успешном ответе агента поле ввода и кнопка отправки разблокируются.
5. Если ответ не получен в пределах таймаута, пользователю показывается ошибка и действие «Перезапустить сессию».
6. Повторная отправка того же сообщения во время ожидания не выполняется (антидубль на UI).
