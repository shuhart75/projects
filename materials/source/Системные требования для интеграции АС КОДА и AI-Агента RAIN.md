**Функциональные требования**
=========================

1. **Общее описание и цели интеграции**
-----------------------------------

### 1.1. Цель

Обеспечить возможность создания бизнес-требований (БТ) в Confluence через AI-агента RAIN из интерфейса АС КОДА после завершения симуляции.

### 1.2. Область ответственности АС КОДА

Обязанности АС КОДА:

* Инициировать диалог с агентом RAIN
* Хранить UI-сессию и состояние диалога (1 час - **уточнить у АС КОДА возможности**)
* Выполнять поверку наличия уже созданного БТ
* Формировать полный контекст запроса
* Обрабатывать HTTP ошибки (все, кроме 200 кода)
* Отображать результат пользователю
* Управлять retry (3 попытки) и восстановлением сессии

### 1.3. Область ответственности агента RAIN

Обязанности агента:

* Работать на основе входного контекста
* Вести диалог с пользователем
* Формировать структуру БТ по шаблону «ПРМ КК»
* Запрашивать недостающие данные
* Создавать страницу в Confluence через API
* Возвращать результат (структура + URL)

### 1.4. Участники и зоны ответственности

| Участник | Роль | Зона ответственности |
| --- | --- | --- |
| **ПРМ аналитик** | Конечный пользователь (роль experiment_editor_CC) | Инициирует сценарий, взаимодействует с агентом, подтверждает создание БТ |
| **АС КОДА** | Клиент | Инициация диалога, передача параметров, отображение результата, хранение сессии UI |
| **AI-агент RAIN** | Сервер | Ведение диалога, генерация БТ, создание страницы в Confluence, предоставление ссылки на БТ |
| **Confluence** | Хранилище | Хранение итоговых страниц БТ по шаблону «ПРМ КК» |
| **СУДИР** | Источник данных | Предоставление ФИО пользователя (через АС КОДА) |

2. **Общие ограничения (Q2)**
-------------------------

Агент RAIN в первой итерации работает со следующими ограничениями:

| Область | Ограничение |
| --- | --- |
| **Хранение сессий** | Сессия не хранится между диалогами агентом. АС КОДА хранит UI-сессию |
| **Продукты** | «Кредитные карты» |
| **ФП Симуляции** | Агент не вызывает ФП самостоятельно. Все данные передает АС КОДА |
| **ФП Риск-параметры** | Только чтение технических имен; русские названия генерируются агентом, но не сохраняются обратно (Q3) |
| **Источник данных** | Данные из ФП Симуляция для продукта «Кредитные карты» |

### 4.3. Детальная логика заполнения «Бизнес-требования» (АС КОДА)

АС КОДА необходимо извлечь из симуляции:

| Данные | | Источник в АС КОДА внутри основной информации по ID Симуляции |
| --- | --- | --- |
| **Дата создания** | | Дате запуска диалогового окна в UI |
| **Наименование запрошенных изменений** | | Дефолтное значение: Изменение Риск-параметров |
| **Бизнес-эффект** | | Если в АС КОДА нет нужного поля, то предложить агенту запросить у пользователя |
| **Отв. аналитик** | | ФИО из СУДИР |
| **Бизнес-требования** | |  |
|  | **Название риск-параметра** | Вкладка «Изменение Риск-параметров» поле «Риск-параметр» |
|  | **AS IS (текущее значение)** | Вкладка «Изменение Риск-параметров» поле «Текущее значение» |
|  | **TO BE (новое значение)** | Вкладка «Изменение Риск-параметров» поле «Новое значение» |

**Правила формирования:**

1. Если изменено несколько риск-параметров - указать все
2. Если значение риск-параметра не изменилось - все равно указать пару AS IS / TO BE (значения совпадают)
3. Если АС КОДА не может заполнить какую-то информацию, то необходимо в запросе указать на необходимость заполнить данные ПРМ аналитику самостоятельно

### 5.2. Первый шаг диалога (вопрос агента)

После нажатия кнопки «Сформировать БТ» и отправки запроса от АС КОДА **агент самостоятельно задает вопрос** (без участия АС КОДА) "Нужно ли создать БТ?".

**Варианты ответа пользователя:**

| Ответ пользователя | Действие агента |
| --- | --- |
| **«Нет»** (или любой отрицательный ответ) | Агент завершает диалог. БТ не создаются. АС КОДА закрывает диалоговое окно. |
| **«Да»** (или любой положительный ответ) | Агент продолжает диалог. Агент **сам дозапрашивает** у АС КОДА данные для формирования БТ. |
| **Агент не получил ответа в течение 1 часа** | АС КОДА хранит сессию 1 час, после 1 часа закрывает диалоговое окно. |

### 5.3. Проверка существующего БТ в АС КОДА

АС КОДА перед открытием диалога проверяет вкладку **«Итоги симуляции»** на наличие поля **«Ссылка на БТ»**.

| Ситуация | Действие АС КОДА |
| --- | --- |
| **Ссылка уже существует** | Во входной запрос от АС КОДА добавляет в поле "bt": "true", в systemPrompt указание для агента: Если bt = true - сообщи пользователю: Бизнес-требование для этой симуляции уже существует. Все равно создать новое? |
| **Пользователь отвечает «Да»** | Агент продолжает создание нового БТ (создается второй документ, ссылка во вкладке "Итоги симуляции" обновляется) |
| **Пользователь отвечает «Нет»** | Агент завершает диалог |

### 5.4. Отображение результата (ссылка на БТ)

После того как агент успешно создал страницу в Confluence и отправил ссылку в диалоговое окно в АС КОДА, АС КОДА должна:

1. Заполнить вкладку **«Итоги симуляции»** поле **«Ссылка на БТ»**
2. Если ссылок несколько (создано несколько БТ для одной симуляции) - отобразить самую последнюю созданную по времени

### 5.5. Кнопка перезапуска сессии

В диалоговом окне **обязательно** должна быть отображена кнопка:

> **«Перезапустить сессию»**

| Условие | Действие |
| --- | --- |
| Агент не отвечает дольше **допустимого времени ожидания** **(значение от Антона, например 30 секунд)** | Пользователь нажимает кнопку «Перезапустить сессию» |
| После нажатия | АС КОДА отправляет новый запрос с теми же параметрами, что были уже ранее заданы в диалоговом окне с новым sessionId |
| Диалог начинается заново (история не сохраняется) | - |

### 5.6. Хранение сессии в UI

* АС КОДА должна держать сессию при бездействии с последнего общения с агентом до 1 часа
* В течение этого часа в диалоговом окне сохраняется диалог (история сообщений)
* ПРМ аналитик не должен заново вести диалог с агентом или повторять введенные данные в рамках 1 часа
* Агент не хранит сессию - после создания БТ и при повторном нажатии кнопки «Сформировать БТ» агент не помнит, что делал предыдущий раз
